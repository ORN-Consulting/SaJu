# tech-stack-decisions.md — 기술 스택 선택 근거

> 사주분석 웹앱의 기술 스택 선택과 그 근거를 기록합니다.

---

## 1. Next.js (App Router)

### 결정

프레임워크로 **Next.js (App Router)**를 사용한다.

### 선택지

| 프레임워크 | 장점 | 단점 |
|-----------|------|------|
| **Next.js (선택)** | SSG 지원, Vercel 최적화, React 생태계 | 프레임워크 크기 |
| Vite + React | 가볍고 빠름 | SSG/SSR 직접 구성 필요 |
| Astro | 정적 사이트에 최적 | React 인터랙션 제한적 |
| SvelteKit | 번들 크기 작음 | React 대비 생태계 작음 |

### 근거

- **SSG(정적 사이트 생성)**: 서버 로직이 없으므로 완전한 정적 사이트로 빌드 가능하다. SEO에도 유리하다.
- **Vercel 네이티브**: Vercel에 배포하면 빌드·CDN·도메인이 자동으로 설정된다.
- **React 생태계**: 차트 라이브러리(Recharts), 애니메이션 등 풍부한 라이브러리 활용 가능하다.
- **확장성**: 추후 회원 기능(API Routes), SEO 페이지(동적 라우팅) 추가 시 프레임워크 교체 없이 확장된다.

## 2. TypeScript (Strict Mode)

### 결정

모든 코드를 **TypeScript strict mode**로 작성한다.

### 근거

- 사주 계산은 천간(10개), 지지(12개), 오행(5개) 등 정해진 값을 다루므로, 타입 시스템이 오류를 사전에 잡아준다.
- 리터럴 유니온 타입으로 천간·지지를 정의하면, 잘못된 값이 들어가는 것을 컴파일 타임에 방지한다.
- 에이전트가 코드를 작성할 때 타입 정보가 가이드 역할을 한다.

### 예시

```typescript
// 잘못된 천간이 들어가면 컴파일 에러
type Stem = '갑' | '을' | '병' | '정' | '무' | '기' | '경' | '신' | '임' | '계';
const stem: Stem = '강'; // ❌ 컴파일 에러
```

## 3. lunar-javascript

### 결정

만세력·음양력·절기 계산에 **lunar-javascript** 라이브러리를 사용한다.

### 선택지

| 방법 | 장점 | 단점 |
|------|------|------|
| **lunar-javascript (선택)** | 절기·음력·천간지지 지원, 활발한 유지보수 | 번들 크기 |
| 직접 구현 | 완전한 통제 | 천문 계산 복잡, 개발 기간 증가 |
| 외부 API | 구현 불필요 | 네트워크 의존, 비용, 지연 |

### 근거

- 절기 시각, 음양력 변환, 간지 계산을 모두 제공한다.
- npm 패키지로 설치 가능하며, 클라이언트 사이드에서 동작한다.
- 외부 API 없이 오프라인에서도 계산 가능하다.
- 활발하게 유지보수되고 있으며, GitHub 스타 수가 높다.

### 주의사항

- 라이브러리의 지원 연도 범위를 확인해야 한다 (일반적으로 1900~2100년).
- 앱의 입력 허용 범위는 1920년~현재 연도로 제한 (birth-input.md 기준).
- 절기 시각의 정확도를 기존 만세력 사이트와 교차 검증해야 한다.
- 번들 크기가 크다면 동적 import나 tree-shaking을 검토한다.

## 4. Tailwind CSS

### 결정

스타일링에 **Tailwind CSS**를 사용한다.

### 근거

- 유틸리티 퍼스트 방식으로 빠른 UI 개발이 가능하다.
- 반응형 디자인이 클래스만으로 처리된다 (`sm:`, `md:`, `lg:`).
- Next.js와의 통합이 기본 지원된다.
- 디자인 시스템 없이도 일관된 스타일을 유지할 수 있다.
- 에이전트가 CSS 파일을 별도 관리하지 않아도 된다.

## 5. Vercel (무료 티어)

### 결정

호스팅에 **Vercel 무료 티어**를 사용한다.

### 선택지

| 플랫폼 | 무료 티어 | Next.js 지원 |
|--------|----------|-------------|
| **Vercel (선택)** | 관대함 | 네이티브 |
| Netlify | 관대함 | 플러그인 필요 |
| Cloudflare Pages | 매우 관대함 | 제한적 |
| GitHub Pages | 정적만 | 없음 |

### 근거

- Next.js를 만든 회사이므로 가장 완벽한 지원을 제공한다.
- Git push만으로 자동 배포된다 (CI/CD 설정 불필요).
- 무료 티어로 월 100GB 대역폭, 자동 HTTPS, 글로벌 CDN을 제공한다.
- 서버 사이드 로직이 없으므로 무료 티어 제한에 거의 도달하지 않는다.

## 6. DB 없음 (의도적 선택)

### 결정

MVP에서는 **DB를 사용하지 않는다**.

### 근거

- 비회원 일회성 계산이므로 저장할 데이터가 없다.
- DB가 없으면 운영 비용, 보안 부담, 개인정보 이슈가 모두 사라진다.
- 완전한 정적 사이트가 가능하므로 성능도 최적이다.

### 확장 계획

- 회원 기능이 필요해지면 **Supabase**를 도입한다 (무료 티어, PostgreSQL, Auth 기본 제공).
- Supabase 선택 근거: Vercel과 통합 용이, 무료 티어 관대, Firebase 대비 SQL 지원.

## 7. 상태 관리: React useState

### 결정

상태 관리에 별도 라이브러리 없이 **React의 useState/useReducer**만 사용한다.

### 근거

- 앱의 상태가 단순하다: 입력값 → 계산 → 결과. 한 방향 흐름이다.
- 전역 상태가 필요한 경우가 없다 (로그인, 장바구니 등 없음).
- Zustand, Redux 등은 이 규모에서 오버엔지니어링이다.
- 추후 상태가 복잡해지면 그때 Zustand를 도입한다.

---

## 8. 차트: Recharts (또는 SVG 직접)

### 결정

오행 레이더 차트에 **Recharts** 라이브러리를 우선 검토하고, 커스터마이징이 부족하면 SVG를 직접 구현한다.

### 근거

- Recharts는 React 친화적이고 RadarChart를 기본 제공한다.
- 오행 색상 커스터마이징이 가능하다.
- 만약 디자인 요구사항이 Recharts로 구현 불가능하면, SVG 직접 구현으로 전환한다.
- D3.js는 이 규모에서 과도하다.

### 에이전트 재량

- Recharts가 적합하면 Recharts 사용
- 커스텀이 필요하면 SVG 직접 구현
- 에이전트가 상황에 따라 판단 가능 (AGENTS.md 자율 범위)

# ui-design-decisions.md — UI/UX 설계 결정

> 사주분석 웹앱의 UI/UX 설계 결정과 그 근거를 기록합니다.

---

## 1. 단일 페이지 구조

### 결정

앱 전체를 **단일 페이지**로 구성한다. 랜딩 → 입력 → 결과가 하나의 페이지에 있다.

### 근거

- 사용자 흐름이 단순하다: 입력 → 결과. 페이지 전환이 필요 없다.
- URL 관리가 간단하다 (라우팅 복잡도 제거).
- 스크롤 기반 인터랙션으로 자연스러운 흐름을 만들 수 있다.
- 모바일에서 페이지 전환보다 스크롤이 더 자연스럽다.

### 확장 시 고려

- SEO 페이지(사주란?, 오행 알아보기)는 별도 라우트로 추가한다.
- 궁합 기능 추가 시에도 같은 페이지 패턴(입력 → 결과)을 재사용한다.

## 2. 사주 카드 형태 (4장)

### 결정

사주팔자를 **4장의 카드**로 시각화한다. 전통 사주 표(세로 4열) 대신 카드를 선택했다.

### 선택지

| 방식 | 장점 | 단점 |
|------|------|------|
| 전통 표 (4열 2행) | 명리학 전공자에게 익숙 | 초보자에게 낯설고, 정보 밀도 낮음 |
| **카드 (선택)** | 각 기둥에 풍부한 정보 담기 가능, 모바일 친화 | 전통과 다름 |
| 가로 한 줄 | 간결 | 정보 표시 공간 부족 |

### 근거

- 카드 안에 천간·지지·오행 색상·십성을 모두 담을 수 있다.
- 모바일에서 2×2 그리드로 자연스럽게 전환된다.
- 시각적으로 구분이 명확하여 초보자도 직관적으로 이해한다.
- 일간(나) 카드를 강조하여 기준점을 명확히 전달한다.

### 카드 배치 순서

**시주 | 일주 | 월주 | 년주** (왼쪽부터)

이는 전통 명리학의 우→좌 배치를 화면의 좌→우로 대응시킨 것이다.

## 3. 오행 차트: 레이더(방사형)

### 결정

오행 분포를 **레이더/방사형 차트**로 시각화한다.

### 선택지

| 차트 유형 | 장점 | 단점 |
|----------|------|------|
| 막대 차트 | 직관적, 구현 쉬움 | 균형/불균형 표현이 약함 |
| **레이더 (선택)** | 5개 오행의 균형을 한눈에 파악 | 축 해석이 필요 |
| 원형 차트 | 비율 표현 명확 | 5개 항목에서 차이 구분 어려움 |
| 아이콘 + 숫자 | 가장 단순 | 시각적 임팩트 부족 |

### 근거

- 오행의 핵심은 **균형**이다. 레이더 차트는 균형/불균형을 시각적으로 가장 잘 보여준다.
- 5개 축이 정확히 오행의 5가지와 매칭된다.
- 차트 아래에 텍스트 보충 정보를 함께 제공하므로, 차트를 못 읽어도 내용 파악이 가능하다.

## 4. 오행 색상 체계

### 결정

오행 각각에 **고유 색상**을 부여하여 앱 전체에서 일관되게 사용한다.

| 오행 | 색상 | 근거 |
|------|------|------|
| 목(木) | 초록 | 나무, 성장의 전통적 상징색 |
| 화(火) | 빨강 | 불의 전통적 상징색 |
| 토(土) | 노랑/갈색 | 흙의 전통적 상징색 |
| 금(金) | 회색/은색 | 금속의 색. 흰색은 배경과 구분 어려워 회색 사용 |
| 수(水) | 파랑 | 물의 전통적 상징색. 검정 대신 파랑 사용 (가독성) |

### 설계 원칙

- 색상만으로 정보를 전달하지 않는다 → 항상 텍스트 라벨 병기 (색각 이상 대응).
- 배경색으로 사용할 때는 연한 버전, 텍스트나 배지에는 진한 버전을 사용한다.
- Tailwind CSS 색상 팔레트 기준으로 매핑하되, 최종 HEX는 디자인 단계에서 확정한다.

## 5. 해석 텍스트 문체: 혼합

### 결정

제목/레이블은 **격식체**, 설명/해석은 **친근체**를 사용한다.

### 예시

```
[격식체] 오행(五行) 분석 결과       ← 섹션 제목
[친근체] 목(木)의 기운이 강해요.     ← 해석 본문
         큰 나무처럼 곧고 강직한
         성격을 가지고 있어요.
```

### 근거

- 격식체 제목은 전문성과 신뢰감을 준다.
- 친근체 해석은 명리학 비전문가도 부담 없이 읽을 수 있다.
- 사주 앱의 주 사용자층이 일반인이므로, 과도한 전문용어는 지양한다.
- 전문 용어 사용 시 반드시 한글(한자) 병기: "용신(用神)"

## 6. 애니메이션 전략

### 결정

**섹션별 순차 페이드인** + **카드 개별 순차 등장** 방식을 사용한다.

### 원칙

| 원칙 | 설명 |
|------|------|
| **절제** | 모던하고 깨끗한 느낌 유지. 과도한 애니메이션 금지 |
| **의미** | 장식이 아닌 정보 전달 흐름을 위한 애니메이션 |
| **접근성** | `prefers-reduced-motion` 미디어 쿼리로 비활성화 가능 |
| **성능** | `transform`과 `opacity`만 사용 (GPU 가속, 리플로우 방지) |

### 타이밍 기준

- 섹션 간 딜레이: 200ms
- 카드 간 딜레이: 100ms
- 개별 애니메이션 duration: 500ms
- easing: ease-out

## 7. 반응형 전략

### 결정

**모바일 퍼스트**로 설계하고, 태블릿·데스크톱으로 확장한다.

### 브레이크포인트

| 구간 | 너비 | Tailwind | 주요 변경 |
|------|------|----------|----------|
| 모바일 | < 640px | 기본 | 1열 배치, 카드 2×2 |
| 태블릿 | 640~1024px | `sm:`, `md:` | 카드 4열, 콘텐츠 확장 |
| 데스크톱 | > 1024px | `lg:` | 최대 너비 제한, 여백 확대 |

### 근거

- 사주 앱의 주 사용 기기는 모바일이 될 가능성이 높다 (가벼운 재미 성격).
- 모바일 퍼스트로 설계하면 핵심 콘텐츠만 남기는 효과가 있다.
- 데스크톱은 여백을 활용하여 여유 있는 레이아웃을 제공한다.

## 8. 에러 표시: 인라인 방식

### 결정

에러 메시지를 **해당 입력 필드 바로 아래에 인라인으로 표시**한다.

### 선택지

| 방식 | 장점 | 단점 |
|------|------|------|
| **인라인 (선택)** | 어디가 문제인지 바로 파악 | 레이아웃 시프트 가능 |
| 토스트 | 레이아웃 영향 없음 | 어떤 필드가 문제인지 불명확 |
| 모달 | 주의 환기 강함 | 과도함, UX 방해 |

### 근거

- 입력 항목이 5~6개로 적으므로 인라인이 가장 직관적이다.
- 어떤 필드에 문제가 있는지 한눈에 알 수 있다.
- 레이아웃 시프트는 에러 메시지 공간을 미리 확보하거나, min-height로 최소화한다.
